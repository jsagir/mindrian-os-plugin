## PR Review Checklist for Moat Assessment

Before merging any PR, reviewers must answer these questions:

### Layer Touch Check

- [ ] Which MWP layers does this PR touch? (list by number 1-7)
- [ ] Does the change strengthen integration BETWEEN layers?
- [ ] Does the change add a new edge type or enrich an existing one?

### Moat Deepening Check

- [ ] Does this make the system harder to replicate? How?
- [ ] Does this create compounding value (gets better with more usage)?
- [ ] Does this leverage Brain IP that competitors cannot access?
- [ ] Does this improve teaching calibration fidelity?

### Surface Area Warning Signs

- [ ] Is this feature standalone (not connected to the cascade)?
- [ ] Could this be added to any folder-based system without modification?
- [ ] Does this work without KuzuDB edges?
- [ ] Is this just a UI addition without intelligence underneath?

If 3+ warning signs are checked, the PR adds surface area. It should either be redesigned to integrate with MWP layers or documented as intentional surface expansion with a plan for future integration.

### Compounding Value Check

Every week of operation should deepen the moat:
- Every user decision (APPROVE/REJECT/DEFER) becomes graph data
- Every HSI computation refines innovation connections
- Every Brain query enriches local KuzuDB
- Every spectral profile builds the room's thinking-mode map
- Every session log feeds future KAIROS dream cycles

Does this PR contribute to any of these compounding loops?

## Anti-Patterns

**1. Feature Island:** A command or agent that operates independently of the cascade pipeline. If it does not generate edges or consume edges, it is an island.

**2. Shallow Integration:** A feature that touches one layer but does not interact with others. Example: a UI component that reads STATE.md but does not trigger proactive intelligence.

**3. Commodity Feature:** A feature that any AI coding plugin could implement identically. Example: syntax highlighting, file search, generic chat. These are fine as utilities but must not consume roadmap priority.

**4. Brain Bypass:** A feature that provides the same quality output with or without Brain connection. Brain enrichment should create a measurable quality differential.
