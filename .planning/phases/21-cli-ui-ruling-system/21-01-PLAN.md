Create `skills/ui-system/SKILL.md` -- the comprehensive rendering rules for ALL MindrianOS terminal output. This file IS the spec AND the implementation (prompt engineering). Follow the YAML frontmatter pattern from existing skills (name, description).

The SKILL.md must contain these sections, implementing all 26 decisions from CONTEXT.md:

**1. Preamble** -- what this skill does, when it activates (always), relationship to other skills

**2. Four-Zone Output Anatomy (per D-03, D-04, D-05, D-26)**
- Zone 1: Header Panel -- room context, venture stage, section name. Standard card format using ASCII box chars (`╭─╮│╰─╯`). Show room name as canary (future multi-room prep).
- Zone 2: Content Body -- varies by body shape (see section 3). This is where the command's payload goes.
- Zone 3: Intelligence Strip -- inline signals from room-proactive. Only appears when HIGH/MEDIUM signals detected. Max 3 signals per screen. Format: `⚠ [contradiction] | ⬜ [gap] | ⚡ [convergence]` with one-line descriptions.
- Zone 4: Action Footer -- NEVER omitted (per D-04). Exactly 2-3 grounded next steps as copy-paste `/mos:` commands. Use `▶` for primary action, `▷` for alternatives.
- Density adaptation (per D-26): If output exceeds 30 lines, compress Zone 1 to one line, show only top 2 signals in Zone 3. Zone 4 stays at exactly 3 always. Short output (<10 lines) is fine -- don't pad.

**3. Five Body Shapes (per D-06 through D-10)**
For each shape, provide: (a) which commands use it, (b) exact rendering template with ASCII example, (c) data source.

- Shape A: Mondrian Board -- progress bars per section using `■` fill chars, 10-char wide bars. Used by `/mos:status`.
- Shape B: Semantic Tree -- folder tree with meaning using `▼▶▷├─└─` symbols. Used by `/mos:tree`, `/mos:room overview`.
- Shape C: Room Card -- wiki-style card with graph relationships (INFORMS, CONTRADICTS, CONVERGES, FILED_FROM edges), floating signal badge. Used by `/mos:room [section]`.
- Shape D: Document View -- content + LazyGraph edges. Used by `/mos:open [artifact]`.
- Shape E: Action Report -- before/after delta with `[N->N]` change indicators. Used by `/mos:act`, `/mos:file-meeting`.

**4. Command-to-Shape Mapping Table (per D-22)**
A reference table mapping every `/mos:` command to its body shape. If a command is not in the table, it defaults to Shape B (Semantic Tree) for structural output or no shape for conversational responses. Include ALL known commands.

**5. Symbol Vocabulary (per D-11, D-12)**
Exactly 12 glyphs with ONE meaning each:
`■` progress fill, `▼` expanded, `▶` collapsed+content / next action, `▷` collapsed+empty / alt action, `├─` not-last item, `└─` last item, `✓` complete, `•` draft, `⚠` warning, `⚡` convergence, `⬜` gap, `→` inline suggestion.
NO EMOJI. NO `🚀✨💫🧠`. EVER. (per D-12)

**6. Color Contract (per D-13, D-14)**
Exactly 5 ANSI colors with semantic meaning:
- Green: success, active, complete
- Cyan: commands, paths, links
- Yellow: warnings, caution
- Red: errors only
- Gray: meta information, timestamps
- Default/White: primary content
- Bold: emphasis
Color is NEVER decoration. When in doubt, use default.

**7. Session Start Contract (per D-15, D-16, D-17)**
Three variants with exact templates:
- Cold (no room): Brief greeting + `/mos:new-project` suggestion
- Warm (room exists, no signals): "Reading the Room" blockquote trace + room summary + suggested action
- Warm+Signals (room with active signals): "Reading the Room" trace + max 2 signals (per D-16) + suggested action
Never show same signal two sessions in a row unless something changed (per D-16).
Always include Larry's "Reading the Room" thinking trace in blockquote format (per D-17).

**8. CLI Voice Rules (per D-23)**
- Terse, structural, confident, action-oriented
- Banned phrases: "Great question!", "I'd be happy to help", "It's important to note", "Let me explain", sentences starting with "I"
- Reference larry-personality/SKILL.md for full voice DNA -- this extends it for terminal context

**9. Error Handling (per D-24)**
Strict three-line pattern:
```
✗ [What failed]
  Why: [reason]
  Fix: /mos:command
```
Never show stack traces. Never show raw error objects.

**10. Help System Format (per D-25)**
`/mos:help` shows commands grouped by flow (Getting started, Working, Reviewing, Brain, Admin).
`/mos:help <command>` shows: 1 description line + 3 examples max. tldr-style, NOT man-page.

**11. Cross-Surface Adaptation (per D-18, D-19)**
- CLI: master template (all ASCII box chars, progress bars, tree symbols)
- Desktop: degrade to bold markdown headers + sentence summaries + blockquote callouts (no box chars)
- Cowork: same box chars as CLI
- 80-column default, 62 content chars. Narrow (<60 cols) compresses headers, collapses trees, limits signals to 1. Wide (>100 cols) does NOT expand.

**12. Dual Context (per D-20, D-21)**
- Every room section folder gets both STATE.md and MINTO.md
- Commands read BOTH before routing
- A section with broken MINTO is worse than a sparse section with solid reasoning
- The "Reasoning" column in status output comes from MINTO health
- Reference the MINTO.md template at `references/templates/MINTO.md`

**CRITICAL CONSTRAINTS:**
- The file must be SELF-CONTAINED. Larry reads this one file and knows how to render every command's output.
- Include ASCII art examples for each body shape so the executor has exact templates to follow.
- Do NOT reference external rendering scripts (per deferred decisions -- keep inline).
- Follow the existing SKILL.md pattern (YAML frontmatter + markdown body).
