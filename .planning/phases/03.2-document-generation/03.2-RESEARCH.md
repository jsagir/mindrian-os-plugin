# Phase 3.2: Document Generation - Research

**Researched:** 2026-03-22
**Domain:** PDF generation via WeasyPrint + Jinja2, CSS paged media, font bundling
**Confidence:** HIGH

## Summary

This phase adds professional De Stijl PDF export to MindrianOS. The pipeline is: room markdown artifacts (YAML frontmatter) -> markdown2 (parse to HTML + extract metadata) -> Jinja2 (inject into document-type template) -> De Stijl CSS stylesheet -> WeasyPrint (render to PDF) -> PyMuPDF (optional TOC bookmarks). All five dependencies are already installed and verified: WeasyPrint 67.0, markdown2 2.5.4, Jinja2 3.1.6, PyMuPDF 1.26.6, PyYAML 6.0.3.

The architecture reference (`references/document-generation.md`) is comprehensive and prescriptive. It defines 7 document types, the full CSS stylesheet with @page rules, section-to-document-to-color mappings, file structure, and command UX. The research validates this spec against WeasyPrint's actual capabilities and identifies implementation patterns, font bundling requirements, and pitfalls.

**Primary recommendation:** Build a single `scripts/render-pdf` Python script that loads room content, selects a Jinja2 template by document type, renders via WeasyPrint with bundled fonts and shared CSS, then optionally adds TOC bookmarks via PyMuPDF. Four document types minimum (thesis, summary, report, profile), with deck and brief as stretch.

### Pattern 1: WeasyPrint + Jinja2 PDF Pipeline
**What:** Load room data, render Jinja2 template to HTML string, pass to WeasyPrint with CSS and FontConfiguration
**When to use:** Every PDF export
**Example:**
```python
# Source: WeasyPrint API docs + Jinja2 FileSystemLoader pattern
from jinja2 import Environment, FileSystemLoader
from weasyprint import HTML, CSS
from weasyprint.text.fonts import FontConfiguration

PLUGIN_DIR = Path(__file__).resolve().parent.parent
TEMPLATES_DIR = PLUGIN_DIR / "templates"
ASSETS_DIR = PLUGIN_DIR / "assets"

def render_pdf(doc_type: str, room_data: dict, output_path: Path):
    # 1. Load template
    env = Environment(loader=FileSystemLoader(str(TEMPLATES_DIR)))
    template = env.get_template(f"{doc_type}.html")

    # 2. Render HTML
    html_string = template.render(
        sections=room_data["sections"],
        metadata=room_data["metadata"],
        colors=SECTION_COLORS,
    )

    # 3. Configure fonts
    font_config = FontConfiguration()

    # 4. Load shared CSS
    css = CSS(filename=str(TEMPLATES_DIR / "destijl-base.css"),
              font_config=font_config)

    # 5. Render PDF -- base_url resolves font url() paths
    HTML(string=html_string,
         base_url=str(PLUGIN_DIR)).write_pdf(
        target=str(output_path),
        stylesheets=[css],
        font_config=font_config,
    )
```

### Pattern 2: Room Data Collection
**What:** Walk room/ directory, parse YAML frontmatter from each artifact, group by section
**When to use:** Before rendering any document
**Example:**
```python
import markdown2
import yaml
from pathlib import Path

SECTIONS = [
    "problem-definition", "market-analysis", "solution-design",
    "business-model", "competitive-analysis", "team-execution",
    "legal-ip", "financial-model"
]

def collect_room_data(room_dir: Path) -> dict:
    sections = {}
    for section_name in SECTIONS:
        section_path = room_dir / section_name
        entries = []
        if section_path.is_dir():
            for f in sorted(section_path.glob("*.md")):
                if f.name in ("ROOM.md", "STATE.md"):
                    continue
                text = f.read_text(encoding="utf-8")
                # markdown2 with metadata extra parses YAML frontmatter
                result = markdown2.markdown(text, extras=["metadata", "fenced-code-blocks", "tables"])
                entries.append({
                    "filename": f.name,
                    "metadata": result.metadata or {},
                    "html": str(result),
                    "title": (result.metadata or {}).get("title", f.stem),
                })
        sections[section_name] = entries
    return {"sections": sections, "metadata": {"generated": datetime.now().isoformat()}}
```

### Pattern 3: PyMuPDF TOC Injection
**What:** After WeasyPrint renders PDF, open with PyMuPDF and add structured bookmarks
**When to use:** Multi-section documents (thesis, report) that need navigable TOC
**Example:**
```python
import fitz  # PyMuPDF

def add_toc(pdf_path: Path, toc_entries: list[tuple[int, str, int]]):
    """Add TOC bookmarks to existing PDF.
    toc_entries: [(level, title, page_number), ...]
    """
    doc = fitz.open(str(pdf_path))
    toc = [[level, title, page] for level, title, page in toc_entries]
    doc.set_toc(toc)
    doc.save(str(pdf_path), incremental=True, encryption=0)
    doc.close()
```

**What:** Load bundled TTF fonts using url() relative to base_url
**When to use:** All templates via shared CSS
**Example:**

```css
    font-weight: normal;
    font-style: normal;
}

    font-family: 'Inter';
    src: url('assets/fonts/Inter-Regular.ttf') format('truetype');
    font-weight: normal;
    font-style: normal;
}
@font-face {
    font-family: 'Inter';
    src: url('assets/fonts/Inter-Bold.ttf') format('truetype');
    font-weight: bold;
    font-style: normal;
}
```

### Anti-Patterns to Avoid
- **System font dependency:** Never rely on fonts being installed on the user's system. Always use @font-face with url() pointing to bundled TTFs. Use `base_url` parameter in WeasyPrint to resolve relative paths.
- **local() in @font-face:** WeasyPrint has known issues with `local()` on some platforms. Always use `url()` with relative paths.
- **Implicit page breaks:** Letting WeasyPrint auto-paginate large documents causes unpredictable breaks. Use explicit `break-before: page` / `break-after: page` on section boundaries.
- **Building HTML strings manually:** Never concatenate HTML strings. Always use Jinja2 templates with proper escaping.
- **Global layout on large docs:** For very large documents, explicit page boundaries turn a global layout problem into a series of local ones, making rendering fast and predictable.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Markdown to HTML | Custom parser | markdown2 with `metadata` + `tables` + `fenced-code-blocks` extras | Edge cases in lists, tables, code blocks |
| YAML frontmatter extraction | Regex splitting on `---` | markdown2 `metadata` extra (parses automatically) | Handles edge cases, multi-line values, nested structures |
| PDF page numbering | Manual page number insertion | CSS `counter(page)` / `counter(pages)` in @page margin boxes | WeasyPrint handles this natively in CSS |
| Running headers | Duplicate content on each page | CSS `position: running()` + `element()` or `string()` | CSS paged media standard, handled by WeasyPrint |
| TOC generation | Scan HTML headings manually | PyMuPDF `set_toc()` with section-level entries | Proper PDF bookmark structure with navigation |
| Font subsetting | Strip unused glyphs | WeasyPrint auto-subsets fonts in PDF output | Built-in, reduces file size automatically |
| PDF merging | Write merge logic | PyMuPDF `insert_pdf()` if needed | C-based, fast, handles all PDF features |

**Key insight:** The entire CSS paged media specification is designed for exactly this use case. WeasyPrint implements it. Fighting against CSS paged media by building custom layout logic in Python is the #1 mistake.

## Common Pitfalls

### Pitfall 1: Font path resolution failure
**What goes wrong:** WeasyPrint can't find bundled TTF files, falls back to system default
**Why it happens:** `base_url` not set or set to wrong directory, relative paths in @font-face don't resolve
**How to avoid:** Always pass `base_url=str(PLUGIN_DIR)` to `HTML()` constructor. All `url()` paths in CSS are relative to this. Test with `fc-list` showing NO Bebas Neue (which is our situation -- fonts are NOT system-installed).
**Warning signs:** PDF renders but headings appear in serif/sans-serif default font, not Bebas Neue

### Pitfall 2: Dark background not filling entire page
**What goes wrong:** Dark #0D0D0D background only covers content area, white margins visible
**Why it happens:** CSS `background` on `body` doesn't extend into page margins by default
**How to avoid:** Set `background: #0D0D0D` on the `@page` rule itself, not just `body`. The @page `background` property fills the entire page including margins.
**Warning signs:** White strips around edges of "dark mode" PDF

### Pitfall 3: Content overflow on constrained documents
**What goes wrong:** Venture brief overflows its single page, executive summary exceeds 2 pages
**Why it happens:** Room with many artifacts generates too much content for the constrained format
**How to avoid:** Templates must truncate/summarize. Jinja2 filters (e.g., `|truncate(300)`) and conditional includes (e.g., `{% if sections['market-analysis']|length > 0 %}`) control content volume. Brief and summary templates should limit to key highlights, not dump everything.
**Warning signs:** Multi-page output for a "1-page" document type

### Pitfall 4: break-inside splitting visual cards
**What goes wrong:** A styled section box (with accent bar, background, border) gets split across pages
**Why it happens:** WeasyPrint follows normal flow and breaks anywhere
**How to avoid:** Apply `break-inside: avoid` on section containers, accent-bar boxes, and card elements. Also use `orphans: 3` and `widows: 3` for text paragraphs.
**Warning signs:** Half a colored box on one page, other half on next page

### Pitfall 5: Empty sections generating blank pages
**What goes wrong:** Document includes empty sections with just a heading and no content
**Why it happens:** Template iterates all 8 sections regardless of content
**How to avoid:** Jinja2 conditional: `{% if sections[name]|length > 0 %}...{% endif %}`. Only render sections that have room artifacts.
**Warning signs:** Thesis PDF with 3 filled sections and 5 blank section pages

### Pitfall 6: Running header showing wrong title
**What goes wrong:** All pages show same header, or header is empty
**Why it happens:** CSS `string-set` must be on an element that appears in document flow before the page break
**How to avoid:** Each section's H2 should have `string-set: doctitle content()` so the running header updates per section. Test with multi-section document.
**Warning signs:** Header stuck on first section name, or blank header on all pages

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| reportlab imperative drawing | WeasyPrint CSS paged media | WeasyPrint 50+ (2022) | CSS-native layout instead of coordinate-based drawing |
| wkhtmltopdf (Chrome headless) | WeasyPrint pure Python | WeasyPrint gained maturity | No headless browser dependency, faster, lighter |
| @font-face local() | @font-face url() with base_url | WeasyPrint 52+ | Cross-platform font loading without system install |
| Manual page numbering | CSS counter(page)/counter(pages) | CSS Paged Media Level 3 | Declarative, automatic |

**Deprecated/outdated:**
- WeasyPrint `cairosvg` dependency: removed in v53+, now uses its own SVG renderer
- `page-break-*` properties: still work as aliases, but `break-before`/`break-after`/`break-inside` are the modern CSS names

## Open Questions

1. **Inter font bundling size**
   - What we know: Inter Regular + Bold TTFs are ~300KB each (~650KB total). Bebas Neue is ~50KB.
   - What's unclear: Whether 700KB of bundled fonts is acceptable for a plugin
   - Recommendation: Bundle both. 700KB is trivial. Alternative: use system sans-serif as fallback, but this degrades the De Stijl aesthetic.

2. **PWS Profile data availability**
   - What we know: Profile structure defined in references/pws-profile-generation.md, maps to team-execution section + methodology outputs
   - What's unclear: How much methodology output a typical user will have when they first try export
   - Recommendation: Profile template must gracefully handle sparse data -- show what exists, leave placeholder sections for missing methodology outputs

3. **Pitch deck and venture brief complexity**
   - What we know: Deck needs landscape @page, brief needs strict 1-page constraint
   - What's unclear: Whether these are worth the complexity for MVP vs thesis/summary/report/profile
   - Recommendation: Implement thesis, summary, report, profile as core 4. Deck and brief as stretch if time allows.
