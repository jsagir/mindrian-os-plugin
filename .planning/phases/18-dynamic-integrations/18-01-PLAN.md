Create `lib/core/integration-registry.cjs` — zero npm dependencies, pure Node.js built-ins only (continues Phase 10 pattern).

The module exports:

1. `INTEGRATION_CATALOG` — object defining all known integrations:
   - `brain`: env=MINDRIAN_BRAIN_KEY, mcp=mindrian-brain, benefit="Enhanced framework suggestions, grading calibration, cross-domain connections", setup="/mos:setup brain"
   - `velma`: env=MODULATE_API_KEY, benefit="Audio transcription with speaker diarization and emotion detection", setup="/mos:setup transcription"
   - `obsidian`: detect=check for .obsidian/ in workspace or parent dirs, benefit="Vault sync — room sections as Obsidian pages with graph view", setup="/mos:setup obsidian" (future)
   - `notion`: env=NOTION_API_KEY or mcp=notion, benefit="Workspace sync — room intelligence in Notion pages", setup="/mos:setup notion" (future)
   - `meeting_source`: mcp keys containing read-ai/vexa/recall-ai, benefit="Auto-fetch latest meeting transcripts", setup="/mos:setup meetings"

2. `detectIntegrations(options)` — scans environment and workspace, returns:
   ```json
   {
     "brain": { "status": "connected|not-configured", "env_set": true|false, "mcp_available": true|false },
     "velma": { "status": "connected|not-configured", "env_set": true|false },
     "obsidian": { "status": "detected|not-configured", "vault_found": true|false },
     "notion": { "status": "connected|not-configured", "env_set": true|false, "mcp_available": true|false },
     "meeting_source": { "status": "connected|not-configured", "provider": "read-ai|vexa|recall-ai|none" }
   }
   ```
   - `options.workDir` defaults to process.cwd()
   - `options.mcpConfig` optional path to .mcp.json (defaults to workDir/.mcp.json)
   - Environment detection: check process.env for each key
   - MCP detection: parse .mcp.json if exists, check mcpServers keys
   - Obsidian detection: walk up from workDir checking for .obsidian/ directory (max 3 levels)
   - Status logic: if env OR mcp detected = "connected", else "not-configured"

3. `checkIntegration(name)` — single integration check, returns its status object.

4. `getContextTriggers(userMessage, roomState)` — returns array of integration suggestions based on context signals. Each suggestion: `{ integration, reason, offer_text }`. Rules:
   - Brain: trigger if user mentions "suggest-next", "grade", "connections", "framework" AND brain not connected
   - Velma: trigger if user mentions "audio", "transcript", "recording", "meeting recording" AND velma not connected
   - Obsidian: trigger if user mentions "notes", "vault", "obsidian", "knowledge base" AND obsidian not detected
   - Notion: trigger if user mentions "notion", "workspace", "wiki" AND notion not connected
   - Meeting source: trigger if user mentions "latest meeting", "auto-fetch", "meeting source" AND no meeting source connected
   - Max 1 suggestion per response (highest relevance). Never suggest during active methodology (roomState.activeMethodology truthy = suppress all).

Create `references/integrations/detection-patterns.md` documenting:
- Each integration's detection method
- Context trigger keywords and phrases
- Offer text templates in Larry's voice (warm, brief, non-pushy)
- Non-blocking rules (when NOT to offer)

Update `skills/larry-personality/SKILL.md` — add a new section "## Integration Offers" after existing sections:

```
## Integration Offers

When Larry detects a task that would benefit from an unconnected integration, he offers it conversationally. This is a PASSIVE behavior — Larry notices context signals and makes a brief, non-pushy mention.

### Rules
- Maximum ONE integration offer per conversation (not per turn — per conversation)
- NEVER offer during an active methodology session (any /mos: command in progress)
- NEVER offer if the user has dismissed it before in this session
- Offer AFTER answering the user's actual question, not before
- Use Larry's natural voice — this is a suggestion, not a prompt

### Offer Pattern
After responding to the user's question, if an integration would clearly help:

> "By the way — [brief benefit]. Want me to set that up? `/mos:setup [integration]`"

### Examples
- User asks for framework suggestions without Brain: "By the way — I'd be sharper with my teaching graph connected. One command: `/mos:setup brain`"
- User pastes audio file path without Velma: "I can transcribe that with speaker identification if you connect Velma — `/mos:setup transcription`"
- User mentions Obsidian vault: "I noticed you have an Obsidian vault. Room sections could sync there — that's on the roadmap."
- User mentions Notion: "Notion sync is coming. For now, `/mos:export` gets your room into any format."

### What NOT to Do
- Do not offer Brain during grading (user chose to grade without it — respect that)
- Do not mention pricing or tiers
- Do not repeat an offer the user ignored
- Do not interrupt methodology output with integration suggestions
```

Update `skills/brain-connector/SKILL.md` — add a subsection under "## Detection" called "### Offer-to-Setup":

```
### Offer-to-Setup

When Brain detection fails (all 3 checks return false) AND the user's request would benefit from Brain enrichment (framework queries, grading, cross-domain connections):

Brain-beneficial signals: mentions of "grade", "connections", "framework recommendation", "what should I use", "suggest-next", "research [topic]"
```
