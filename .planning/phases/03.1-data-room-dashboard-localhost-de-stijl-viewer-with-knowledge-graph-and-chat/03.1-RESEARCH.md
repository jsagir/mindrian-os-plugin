# Phase 3.1: Data Room Dashboard - Research

**Researched:** 2026-03-22
**Domain:** Localhost dashboard (Cytoscape.js graph + De Stijl + chat), bash tooling, zero-dependency architecture
**Confidence:** HIGH

## Summary

Phase 3.1 delivers a visual front door to the Data Room: a single-HTML-file localhost dashboard showing the knowledge graph with De Stijl styling and a chat box for querying room intelligence. The architecture is deliberately minimal -- a bash script generates `graph.json` from `room/` contents, a Python `http.server` serves a single `dashboard.html` file, and the browser opens automatically. Everything is self-contained with zero npm/pip installs.

The hardest design decision is the chat box. After researching options, the recommended approach is a **pre-computed intelligence model**: the `build-graph` script runs `analyze-room` and embeds its GAP/CONVERGE/CONTRADICT findings directly into the graph JSON. The chat box uses client-side JavaScript to search this embedded data and answer natural language queries like "what gaps do I have?" or "show contradictions" without any server round-trip or LLM call. This keeps the dashboard fully private, zero-dependency, and instantly responsive. Users who want deeper AI-powered querying can use `/mindrian-os:room` in their Claude session.

**Primary recommendation:** Single HTML file with inline CSS/JS + CDN links (Cytoscape.js, Google Fonts). Graph data embedded as `<script>` JSON block. Served via `python3 -m http.server`. Chat is client-side keyword search over pre-computed room intelligence.

### Pattern 2: Graph Data Generation Script (build-graph)
**What:** Bash script that reads `room/` directory, parses YAML frontmatter, runs `analyze-room`, and outputs `dashboard/graph.json`.
**When to use:** Every time the user runs `/mindrian-os:room view`.
**Key logic:**
```bash
#!/usr/bin/env bash
# 1. Scan room/*/*.md files (excluding ROOM.md, STATE.md)
# 2. For each artifact: extract frontmatter (methodology, room_section, pipeline, pipeline_stage, pipeline_input, created)
# 3. Create a node for each artifact: { data: { id, label, section, methodology, created } }
# 4. Create section group nodes (8 parent nodes, one per section)
# 5. Edges from pipeline provenance: FEEDS_INTO (pipeline_stage N -> N+1 in same pipeline)
# 6. Edges from analyze-room output: CONTRADICTS (CONTRADICT: lines), CONVERGES (CONVERGE: lines)
# 7. Edges from cross-references: INFORMS (artifacts referencing [[other-section]])
# 8. Include intelligence data for chat: gaps, convergence themes, contradictions
# 9. Output JSON to dashboard/graph.json
```

### Pattern 3: Pre-Computed Chat Intelligence

### Pattern 4: Serve-Dashboard Script with Port Finding
**What:** Bash script that generates graph data, finds an available port, starts Python server, opens browser.
**Key logic:**
```bash
#!/usr/bin/env bash
# 1. Run build-graph to generate fresh graph.json
# 2. Find available port (start at 8420, increment if busy)
# 3. Start python3 -m http.server $PORT in dashboard/ directory (background)
# 4. Open browser: xdg-open (Linux), open (macOS), cmd.exe /c start (Windows/WSL)
# 5. Print URL to stdout for Claude to relay to user
# 6. Trap EXIT to kill server on script termination
```

### Anti-Patterns to Avoid
- **WebSocket chat to Claude CLI:** Massively over-engineered for v1. Would require persistent Node server, WebSocket protocol, Claude CLI stdin piping, async response handling. Save for v2.
- **npm install anything:** Breaks zero-dependency constraint. CDN links only.
- **Generating HTML dynamically:** Template the HTML once, load data via JSON fetch. Simpler to maintain.
- **Auto-refresh via file watcher:** Adds complexity (fs.watch, polling). A manual refresh button that re-fetches `graph.json` is sufficient for v1.
- **Embedded graph.json in HTML:** Makes the HTML file huge and non-cacheable. Separate JSON file is cleaner and allows refresh without regenerating HTML.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Graph layout algorithm | Custom force-directed physics | Cytoscape.js CoSE layout | Graph layout is deceptively complex; CoSE handles node overlap, edge routing, compound nodes |
| YAML frontmatter parsing in bash | Full YAML parser | `grep`/`sed` for known fields | Our frontmatter is simple key-value; full YAML parsing in bash is fragile. Match only the 6 known fields |
| Localhost server | Custom HTTP handler | `python3 -m http.server` | One line, zero install, serves static files correctly |
| Cross-platform browser open | Custom detection | `xdg-open` / `open` / `cmd.exe /c start` with OS detection | Well-known pattern, 3 lines of bash |
| Knowledge graph data model | Custom graph format | Cytoscape.js elements format directly | Generate the exact format Cytoscape expects; no intermediate format needed |

**Key insight:** The entire dashboard is a visualization layer over data that already exists (room artifacts + analyze-room output). The scripts transform existing data into Cytoscape format. No new intelligence or computation is needed beyond what `analyze-room` and `compute-state` already provide.

## Common Pitfalls

### Pitfall 1: Port Conflicts
**What goes wrong:** Python http.server fails because port 8000 is already in use (another dev server, etc.)
**Why it happens:** Hard-coded port numbers on developer machines
**How to avoid:** Start at a distinctive port (8420), check if busy, increment. Also check for existing MindrianOS dashboard processes and kill/reuse.
**Warning signs:** "Address already in use" error

### Pitfall 2: CORS / Fetch Issues with file:// Protocol
**What goes wrong:** Opening `index.html` directly (file://) instead of via http server causes fetch('graph.json') to fail with CORS error
**Why it happens:** Browsers block cross-origin requests from file:// protocol
**How to avoid:** Always serve via http.server. Never instruct users to open the HTML file directly. The serve-dashboard script handles this.
**Warning signs:** Console errors about CORS, graph not loading

### Pitfall 3: Cytoscape.js Node Overlap with Few Nodes
**What goes wrong:** CoSE layout looks bad with < 5 nodes (everything clumps together or spreads too far)
**Why it happens:** Force-directed layouts need minimum node count for good spacing
**How to avoid:** Use different layout parameters based on node count. For < 5 nodes, use `grid` or `circle` layout. For 5+, use `cose` with tuned `nodeRepulsion: 8000` and `idealEdgeLength: 120`.
**Warning signs:** Nodes stacked on top of each other, or single node in corner

### Pitfall 4: Empty Room Renders Blank Graph
**What goes wrong:** User runs `/mindrian-os:room view` with an empty or near-empty room, sees blank white (dark) screen
**Why it happens:** No artifacts yet = no nodes = empty graph
**How to avoid:** Always render the 8 section nodes as the base graph (like an empty constellation). Show them as placeholder/muted nodes. Artifacts appear as child nodes within their section. Even an empty room shows the structure.
**Warning signs:** Blank graph container

### Pitfall 5: De Stijl Chromatic Density Violation
**What goes wrong:** All 8 section colors visible simultaneously on the knowledge graph, violating the "max 3 chromatic hues per screen" rule
**Why it happens:** Graph shows all sections at once
**How to avoid:** This is a documented exception -- the graph is a data visualization, not a UI panel. Use section colors at reduced opacity (40-60%) for inactive/background nodes, full opacity only for selected/hovered node and its neighbors. The 3-hue rule applies to chrome, not data visualization.
**Warning signs:** Visual noise, colors competing for attention

### Pitfall 6: WSL Browser Opening
**What goes wrong:** `xdg-open` doesn't work in WSL because there's no Linux desktop environment
**Why it happens:** WSL runs Linux userland but the browser is on the Windows side
**How to avoid:** Detect WSL (check `/proc/version` for "microsoft" or "WSL"), use `cmd.exe /c start` or `wslview` instead of `xdg-open`. The current dev environment IS WSL, so this is critical.
**Warning signs:** "xdg-open: no method available for opening" error

### YAML Frontmatter Extraction in Bash
```bash
# Source: MindrianOS existing pattern (analyze-room, compute-state)
# Extract known frontmatter fields from a room artifact
extract_frontmatter() {
  local file="$1"
  local in_frontmatter=false
  local line_num=0

  while IFS= read -r line; do
    line_num=$((line_num + 1))
    if [ "$line_num" -eq 1 ] && [ "$line" = "---" ]; then
      in_frontmatter=true
      continue
    fi
    if $in_frontmatter && [ "$line" = "---" ]; then
      break
    fi
    if $in_frontmatter; then
      echo "$line"
    fi
  done < "$file"
}

# Usage: extract specific field
get_field() {
  local file="$1"
  local field="$2"
  extract_frontmatter "$file" | grep "^${field}:" | sed "s/${field}:[[:space:]]*//" | tr -d '"'
}

# Example:
# methodology=$(get_field "room/problem-definition/domain-exploration.md" "methodology")
# pipeline=$(get_field "room/problem-definition/domain-exploration.md" "pipeline")
```

open_browser() {
  local url="$1"
  if grep -qi microsoft /proc/version 2>/dev/null; then
    # WSL
    cmd.exe /c start "" "$url" 2>/dev/null
  elif [ "$(uname)" = "Darwin" ]; then

### Chat Box Query Matching (Client-Side)
```javascript
// Source: custom pattern for MindrianOS dashboard
function handleChatQuery(query, intelligence) {
  const q = query.toLowerCase().trim();

  // Direct command queries
  if (/^(gaps?|missing|what.*(need|missing|empty))/.test(q)) {
    return formatGaps(intelligence.gaps);
  }
  if (/^(contradict|conflict|mismatch|inconsisten)/.test(q)) {
    return formatContradictions(intelligence.contradictions);
  }
  if (/^(converg|theme|pattern|common|shared)/.test(q)) {
    return formatConvergence(intelligence.convergence);
  }
  if (/^(summar|overview|status|how.*(doing|look|room))/.test(q)) {
    return formatSummary(intelligence.summary);
  }

  // Section-specific queries
  for (const [section, color] of Object.entries(SECTION_COLORS)) {
    const sectionWords = section.replace('-', ' ').split(' ');
    if (sectionWords.some(w => q.includes(w))) {
      return formatSection(intelligence, section);
    }
  }

  // Edge type queries
  if (/feeds|chain|pipeline|flow/.test(q)) {
    return formatEdgeType(intelligence, 'FEEDS_INTO');
  }

  // Fallback
  return "I can answer about gaps, contradictions, convergence themes, and section status. " +
         "For deeper AI-powered analysis, use /mindrian-os:room in your Claude session.";
}
```

## Open Questions

1. **CDN fallback for offline use**
   - What we know: CDN links require internet. Dashboard is "localhost" but Cytoscape.js loads from unpkg.
   - What's unclear: Should we bundle cytoscape.min.js locally for true offline use?
   - Recommendation: For v1, CDN is fine. Add a fallback comment noting that cytoscape.min.js can be downloaded to `dashboard/lib/` for offline use. Do not over-engineer v1.

2. **Graph data refresh strategy**
   - What we know: `build-graph` runs on every `/mindrian-os:room view` invocation. Dashboard has a refresh button.
   - What's unclear: Should the refresh button re-run `build-graph` (requires server endpoint) or just re-fetch the JSON (requires user to re-run the command)?
   - Recommendation: Refresh button re-fetches `graph.json` from the server. If the user wants new data, they run `/mindrian-os:room view` again (which regenerates graph.json and refreshes the browser). Keep it simple.

3. **Node detail panel**
   - What we know: Clicking a node should show detail. The De Stijl design system has card patterns.
   - What's unclear: How much artifact content to show? Just metadata or full content?
   - Recommendation: Show a slide-in panel (from right, 200ms ease-out, x-axis only) with: artifact title, methodology, created date, section, pipeline provenance. Include a "View in Room" note pointing to the file path. Full content display is out of scope for v1.
