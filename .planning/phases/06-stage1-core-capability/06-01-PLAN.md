Create `references/meeting/` directory with 5 reference files. These are the domain knowledge Larry reads before processing a transcript.

**transcript-patterns.md:**
- Document 5 transcript formats with exact regex-style patterns: Zoom (`[Speaker Name] HH:MM:SS`), Microsoft Teams (`Speaker Name\nHH:MM -- HH:MM`), Otter.ai (`Speaker Name  HH:MM`), Google Meet (`HH:MM:SS Speaker Name`), and Raw Paste (name-colon, quoted speech, paragraph breaks).
- For each format: show example, extraction pattern for speaker name + timestamp + text body.
- Include a "Format Detection" section: Larry should try matching each format's signature pattern in order. If none match, treat as Raw Paste and ask user to identify speaker changes.
- Include guidance for Larry: "If speakers are labeled as 'Speaker 1', 'Speaker 2' etc, ask the user to provide real names and roles before proceeding."

**segment-classification.md:**
- Define 6 segment types with clear boundaries and examples: insight (new idea/finding/connection), advice (recommendation/guidance/lesson), question (open question/challenge/concern), decision (agreement/commitment/choice), action-item (task assignment with owner), noise (small talk/logistics/off-topic/status updates).
- Include classification heuristics: advice usually comes from mentors/advisors, insights from researchers, questions from investors.
- Include a "Filing Priority" section: mentor/investor/customer segments = HIGH, advisor = MEDIUM-HIGH, team-member = MEDIUM, user(self) = LOW. Larry should present HIGH-priority segments first.
- Include ambiguity rules: "If a segment could be advice OR insight, classify based on speaker role. Mentors give advice. Researchers surface insights."

**section-mapping.md:**
- Create a 6x8 matrix: rows are segment_types (insight, advice, question, decision, action-item, noise), columns are 8 room sections. Each cell says whether that combination maps to that section, with conditions.
- Add role-aware overrides: "Mentor advice about market -> market-analysis. Researcher insight about tech -> solution-design. Investor question about competition -> competitive-analysis."
- Include default mappings when role is ambiguous: decision -> problem-definition, action-item -> team-execution, general insight -> problem-definition.

**artifact-template.md:**
- The exact YAML frontmatter template from the research doc, including wicked problem fields: methodology, created, source, speaker, speaker_role, segment_type, confidence, meeting_date, room_section, assumptions (array of {claim, status: unvalidated, impacts: []}), perspective, cascade_sections.
- Body template: title derived from segment content, the speaker's statement (quoted or paraphrased), source attribution line.
- Include instructions for Larry: "For the assumptions field, extract at least one implied assumption from every non-noise segment. If none is obvious, use `assumptions: none_detected`. The assumption is the CLAIM that must be true for this insight to be valid."

**summary-template.md:**
- The exact meeting summary template from the research doc: frontmatter (methodology, created, source, meeting_type, participants, room_section: problem-definition), then sections for Key Decisions, Insights Filed (with count, section, title, speaker, confidence), Assumptions Surfaced, Contradictions Detected, Gaps Identified, Cross-Subsystem Cascades, Action Items (checkboxes with assignee), Rejections (segments skipped with user's stated reason).
- meeting_type options: team-sync, mentor-session, investor-meeting, advisory, customer-interview, research-review.

Create `commands/file-meeting.md` following the thin-command + thick-reference pattern (same structure as beautiful-question.md, setup.md).

**Frontmatter:**
```yaml
---
name: file-meeting
description: File a meeting transcript into your Data Room -- paste text, provide a file, or provide audio that gets transcribed locally
allowed-tools:
  - Read
  - Write
  - Bash
  - Glob
disable-model-invocation: true
---
```

**Setup section:** Read all 5 reference files from references/meeting/. Read references/personality/voice-dna.md for Larry's voice. Read room/STATE.md for venture context (if exists). Check if room/ directory exists -- if not, tell user to run `/mindrian-os:new-project` first.

**Step 1: Get Transcript Input.**
Ask user: "Got a transcript? Paste it here, give me a file path, or drop an audio file."
- If paste: accept the pasted text directly.
- If file path: Read the file using Read tool. Support .txt, .md, .vtt, .srt.
- If audio file (.mp3, .m4a, .wav, .ogg): Check if `scripts/transcribe-audio` exists and is executable. If not, tell user: "Audio transcription isn't set up yet. Run `/mindrian-os:setup transcription` to enable it. For now, paste the text or give me a text file." If script exists, run `bash scripts/transcribe-audio <filepath>` and use the output.
- Also ask: "When was this meeting?" to get meeting_date. "What kind of meeting?" to get meeting_type.

**Step 2: Speaker Identification.**
Using patterns from transcript-patterns.md, identify all unique speakers. Present: "I found N speakers: [list]. Let me know their roles: mentor, researcher, team-member, investor, advisor, or customer. If any names are wrong, correct them now."
Wait for user confirmation. This is BLOCKING -- do not proceed until speakers and roles are confirmed.

**Step 3: Segment Classification.**
For each speaker segment, classify using the taxonomy from segment-classification.md. Do NOT present each classification individually. Instead, produce an internal working list.

**Step 4: Section Mapping + Confirm-Then-File.**
Using the routing matrix from section-mapping.md, map each non-noise segment to a room section. Then present in SECTION BATCHES (not individual segments) to avoid confirmation fatigue:

"I found N non-noise segments. Here's what I'd like to file:

**solution-design** (3 segments):
1. Tyler (researcher, insight): 'Ionic liquids avoid harsh acid...' [HIGH]
2. Tyler (researcher, insight): 'Green chemistry reduces equipment cost...' [MEDIUM]
3. Lawrence (mentor, advice): 'Consider the patent landscape...' [HIGH]

File all 3 to solution-design? [all / review individually / skip section]"

Repeat for each section that has segments. For each confirmed segment, create the artifact using the template from artifact-template.md. Write to `room/{section}/{slug}.md`.

For REJECTED segments: ask "Why skip this one?" and record the reason. This is data, not optional.

For REDIRECTED segments: move to the user's specified section.

**Step 5: Create Meeting Summary.**
After all filings complete, create the summary using summary-template.md. Write to `room/problem-definition/meeting-summary-{date}.md`. Include all 8 summary sections. The Rejections section must list every skipped segment with the user's stated reason.

**Step 6: Cross-Relationship Scan.**
After summary creation, scan the room for the 5 relationship types from CLAUDE.md:
1. INFORMS -- does any new artifact reference content in another section?
2. CONTRADICTS -- does any new artifact conflict with existing claims?
3. CONVERGES -- do themes from this meeting appear in 3+ sections?
4. INVALIDATES -- does any new artifact make an existing assumption stale?
5. ENABLES -- does any new artifact unblock something in another section?

Present findings conversationally: "I noticed [relationship]. Want to explore this?" For CONTRADICTS and INVALIDATES, this is especially important -- these are the wicked problem cascade triggers.

Note: Step 6 is the wicked problem management value. Budget equal conversational effort for Steps 1-5 (filing) and Step 6 (discovery). Do NOT treat Step 6 as an afterthought.

**Update help.md:**
Add file-meeting to the infrastructure commands count (update "6 infrastructure commands" to "7 infrastructure commands" or add to appropriate category). Add an entry in the command recommendations for any stage:
> "Had a meeting? Run `/mindrian-os:file-meeting` -- paste your transcript and I'll file everything to the right rooms with full provenance tracking."
