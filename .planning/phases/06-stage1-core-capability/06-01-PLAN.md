  <action>
Create `references/meeting/` directory with the 3 core analytical reference files. These contain the classification logic and routing rules that the file-meeting command depends on.

**transcript-patterns.md**: Document speaker identification patterns for 6 transcript formats:
- Zoom: `HH:MM:SS Speaker Name: text`
- Teams: `Speaker Name HH:MM:SS` or `[HH:MM:SS] Speaker Name`
- Otter.ai: `Speaker Name  HH:MM` with double-space separator
- Google Meet: `Speaker Name HH:MM:SS`
- Raw paste (no labels): No timestamps, no speaker labels -- Larry must infer from content or ask user
- Velma output: JSON with `speaker_id`, `text`, `start_time`, `end_time`, `emotions[]`
Include regex patterns for each format. Include guidance for Larry when format is ambiguous.

**segment-classification.md**: Define the 6 segment types with priority ordering:
1. decision (HIGHEST) -- explicit choices made, direction set
2. action-item (HIGH) -- tasks assigned, deadlines mentioned
3. insight (MEDIUM) -- new information, discoveries, data points
4. advice (MEDIUM) -- recommendations, suggestions, should/could statements
5. question (LOW) -- open questions, uncertainties raised
6. noise (SKIP) -- greetings, small talk, logistics, off-topic

For each type: definition, example phrases, role-aware heuristics (e.g., investor question about financials = HIGH not LOW), and classification confidence indicators. Include the "flag potential noise" rule: when noise contains a proper noun, competitor name, or number, flag it for user review.

**section-mapping.md**: Create the full routing matrix. 12 speaker roles (mentor, researcher, team-member, investor, advisor, customer, founder, partner, domain-expert, government, competitor, unknown) x 6 segment types x 8 room sections (problem-definition, market-analysis, solution-design, business-model, competitive-analysis, team-execution, legal-ip, financial-model). Include role-weighted priority modifiers (e.g., investor concern about financials = priority boost for financial-model). Not every cell has a mapping -- use "Larry decides based on content" for ambiguous combinations.
  </action>

  </verify>
  <done>3 core analytical reference files exist: transcript patterns with 6 format regexes, segment classification with 6-type taxonomy and role-aware heuristics, section mapping with 12x6x8 routing matrix</done>

Create the 4 template/spec reference files in `references/meeting/`. These define the output formats that the file-meeting command produces.

**artifact-template.md**: Define the wicked-problem-aware YAML frontmatter template:
```yaml
---
methodology: file-meeting
created: {YYYY-MM-DD}
source: transcript
speaker: {name}
speaker_role: {role from 12-type taxonomy}
segment_type: {type from 6-type taxonomy}
confidence: {0.0-1.0}
meeting_date: {YYYY-MM-DD}
meeting_name: {meeting identifier}
room_section: {target section}
assumptions:
  - claim: "{extracted assumption}"
    status: unvalidated
    impacts: [{affected sections}]
perspective: {speaker_role}
cascade_sections: [{sections this insight may affect}]
---
```
Include examples for each segment type. Document the `assumptions: none_detected` fallback. Note the `source: transcript` vs future `source: velma` distinction.

**summary-template.md**: Define the dual-storage meeting summary format:
- Full summary (in `room/meetings/YYYY-MM-DD-{name}/summary.md`): Narrative lead paragraph in Larry's voice, then structured sections: ## Key Decisions, ## Insights Filed ({count} to {sections}), ## Contradictions Detected, ## Gaps Identified, ## Action Items (with owners, no invented deadlines), ## Rejections (with structured reasons), ## Speakers ({count}, with roles)
- Compact reference (at `room/meeting-YYYY-MM-DD-{name}.md`): One-paragraph summary + link to full summary + counts (decisions, insights, actions)
- Include the `room/meetings/YYYY-MM-DD-{name}/transcript.md` format for stored transcripts
- Include the `room/meetings/YYYY-MM-DD-{name}/filed-to/` reference pattern (markdown files with links to filed artifacts)

**speaker-profile-template.md**: Define the ICM nested folder structure for new speakers:
```
room/team/{role-plural}/{firstname-lastname}/
  PROFILE.md
  insights/
  advice/
  connections/
  concerns/
```
Where `{role-plural}` maps: mentor->mentors, researcher->researchers, team-member->members, investor->investors, advisor->advisors, customer->customers, founder->founders, partner->partners, domain-expert->domain-experts, government->government, competitor->competitors, unknown->contacts

Include the PROFILE.md template with YAML frontmatter (name, role, affiliation, first_meeting, meetings_attended, created_by: file-meeting, research_status: pending) and body sections (Context, Expertise, Key Contributions). Note that proactive web research runs AFTER the filing pipeline via scripts/research-speaker, not during it (Pitfall 7 from research).

**live-join-interface.md**: Design spec (NOT implementation) for future external transcript sources:
- `--join <meeting-url>` flag: attend-mcp / Recall.ai / Vexa integration
- `--latest` flag: Read AI MCP pulls most recent transcript
- Document attend-mcp (github.com/xpos/attend-mcp), Recall.ai ($0.50/hr), and Vexa (Apache-2.0, self-hostable) as options
- Describe the interface contract: external source -> same transcript format -> same pipeline
- This file is Phase 6 design-only. Phase 8 implements `--latest`. Phase 9+ implements `--join`.
