**commands/file-meeting.md**: Create the thin command following the established pattern. Frontmatter:
```yaml
---
name: file-meeting
description: File a meeting transcript into your Data Room -- paste text, provide a file, or transcribe audio via Velma
allowed-tools:
  - Read
  - Write
  - Bash
  - Glob
---
```

The command has 6 pipeline steps. Larry (the AI personality) executes each step conversationally:

**Setup section** — Load ALL reference files:
1. Read `references/personality/voice-dna.md` for Larry's voice
2. Read `references/meeting/transcript-patterns.md`
3. Read `references/meeting/segment-classification.md`
4. Read `references/meeting/section-mapping.md`
5. Read `references/meeting/artifact-template.md`
6. Read `references/meeting/summary-template.md`
7. Read `references/meeting/speaker-profile-template.md`
8. Read `references/meeting/cross-relationship-patterns.md` (if file exists -- guard with "Read this file if it exists, skip gracefully if not")
9. Read `room/STATE.md` for venture context (if exists)
10. Scan `room/team/` for known speaker profiles (glob `room/team/*/*/PROFILE.md`)

**Step 1: Get Transcript Input** —
- No flags (default): Ask user to paste transcript text. Accept multi-line paste. Detect format using transcript-patterns.md regex patterns.
- `--file <path>`: Read file at path. Detect format. Support .txt and .md extensions.
- `--audio <path>`: Check if Velma is configured (VELMA_API_KEY in env or .mcp.json). If not, offer inline setup: "Audio needs Velma configured. Set it up now? (30 seconds)" — if yes, run the setup transcription flow; if no, suggest paste/file alternatives. If configured, run `scripts/transcribe-audio <path>` and use output.
- `--latest` / `--join <url>`: Print "Not yet available. Coming in a future update. Use paste, --file, or --audio for now." (Designed in live-join-interface.md, not implemented.)

After getting transcript, infer meeting metadata from content and Data Room context:
- Meeting date (from timestamps, file date, or ask user)
- Meeting purpose (from content themes)
- Participant count
Present inferences: "This looks like a [type] meeting from [date] with [N] participants. Correct?"

**Step 2: Speaker Identification + Profile Creation** —
- Parse speaker labels from transcript using transcript-patterns.md patterns
- If no labels found: Ask user "Who was in this meeting?" and use their response + content analysis to attribute segments
- Cross-reference against `room/team/` profiles:
  - Known match: "Lawrence matches team/mentors/lawrence-aronhime/ (mentor)" — mark AUTO-MATCHED
  - Partial match: "This sounds like Lawrence — mentor from team directory. Confirm?"
  - Unknown: Infer role from content ("Speaker 2 discussed financials and seems senior — could this be your advisor?")
- Present smart hybrid table:
  ```
  | Speaker | Match | Role | Status |
  |---------|-------|------|--------|
  | Lawrence | team/mentors/lawrence-aronhime/ | mentor | AUTO-MATCHED |
  | Tyler | (no match) | researcher? | NEEDS CONFIRMATION |
  ```
- User confirms matches and fills unknowns. Roles from 12-type taxonomy: mentor, researcher, team-member, investor, advisor, customer, founder, partner, domain-expert, government, competitor, unknown
- For CONFIRMED unknown speakers: Run `scripts/create-speaker-profile` to create ICM nested folder profile stub. Track newly created profiles in a list for the post-pipeline research step.
- Emotion signals from Velma (if audio input): Surface only strong emotions (score > 0.7). Example: "Tyler was notably skeptical when discussing market size."

**Step 3: Segment Classification (Priority-First)** —
- Classify each non-greeting segment using segment-classification.md taxonomy
- Apply role-aware heuristics from section-mapping.md (investor question about financials = HIGH priority)
- Sort by priority: decisions (HIGHEST) > action-items (HIGH) > insights (MEDIUM) > advice (MEDIUM) > questions (LOW)
- Flag potential noise that contains proper nouns, competitor names, or numbers for user review
- Show Larry's reasoning for EVERY classification: "This is an insight about market size -> market-analysis. Confidence: 0.85"

**Step 4: Section Mapping + Confirm-Then-File** —
- Present segments batched by section, within each section sorted by priority:
  ```
  **DECISIONS (2 segments):**
  1. Lawrence (mentor): "Focus on B2B first, consumer can wait."
     -> team-execution | decision | HIGH confidence
     Reasoning: Direct strategic direction from mentor

  File both decisions? [all / review individually / skip]
  ```
- For each batch: user can file all, review individually, or skip
- When user rejects: "Why skip? [not relevant] [already known] [wrong section] [other]" — capture structured rejection reason
- For each filed artifact: Create markdown file in the target room section using artifact-template.md frontmatter. Include ALL provenance fields: methodology, created, source, speaker, speaker_role, segment_type, confidence, meeting_date, meeting_name, room_section, assumptions, perspective, cascade_sections.
- Also create a reference copy in the speaker's profile folder (e.g., `room/team/mentors/lawrence-aronhime/insights/2026-03-22-market-focus.md`)
- Use the existing confirm-then-file UX pattern from room-passive

**Step 5: Create Meeting Summary + Archive** —
- Create meeting archive directory: `room/meetings/YYYY-MM-DD-{name}/`
- Store processed transcript: `room/meetings/YYYY-MM-DD-{name}/transcript.md`
- Create full summary following summary-template.md:
  - Narrative lead paragraph in Larry's voice ("Lawrence pushed hard on market validation today. Three things you can't ignore...")
  - ## Key Decisions (with who decided)
  - ## Insights Filed ({count} insights to {sections list})
  - ## Contradictions Detected (if any — skip section if none)
  - ## Gaps Identified (rooms that should have received input but didn't)
  - ## Action Items (with owners from speaker context, deadlines only when explicitly mentioned)
  - ## Rejections (segments rejected with structured reasons — this IS data)
  - ## Speakers ({count} speakers with roles)
- Store at `room/meetings/YYYY-MM-DD-{name}/summary.md`
- Create filed-to/ directory with markdown reference files pointing to each filed artifact
- Create compact root reference: `room/meeting-YYYY-MM-DD-{name}.md` — one paragraph + link to full summary + counts

**Step 6: Cross-Relationship Batch Scan** —
- After ALL filing is complete, scan filed artifacts against existing Data Room content
- Load `references/meeting/cross-relationship-patterns.md` for detection heuristics (already loaded in Setup if available)
- Look for 5 relationship types:
  - INFORMS: new artifact references or provides evidence for another section
  - CONTRADICTS: new artifact conflicts with existing claim
  - CONVERGES: theme from this meeting appears in 3+ sections
  - INVALIDATES: new artifact makes an existing assumption stale
  - ENABLES: new artifact unblocks something in another section
- Surface only significant findings (not every minor cross-reference)
- Present as: "This meeting changed something in your Data Room:" followed by the significant cross-relationships
- If no significant cross-relationships found, say so briefly
- Tier 0 implementation: Larry's conversational reasoning against room content (no LSA/MiniLM in Phase 6 — that's Phase 8)

**Post-Pipeline: Proactive Speaker Research** —
- If new speaker profiles were created in Step 2, run `scripts/research-speaker` for each one
- Present: "I created profiles for {names}. Let me research them online to fill in context..."
- For each new speaker, run: `bash scripts/research-speaker <room_dir> <speaker-slug> <display-name> <project-context>`
- After research completes, present findings to user for confirmation: "Here's what I found about {name}. Want me to update their profile?"
- User confirms before any research content is written to PROFILE.md

End with: "Meeting filed. {N} artifacts across {M} sections. {P} speakers identified. Anything else from this conversation?"

**scripts/create-speaker-profile**: Bash script that creates an ICM nested folder profile.

Input: 4 arguments: $1=room_dir, $2=speaker_name (slug: firstname-lastname), $3=speaker_role, $4=display_name
Output: Creates the folder structure and minimal PROFILE.md

Script:
1. Map role to plural directory: mentor->mentors, researcher->researchers, team-member->members, investor->investors, advisor->advisors, customer->customers, founder->founders, partner->partners, domain-expert->domain-experts, government->government, competitor->competitors, unknown->contacts
2. Create directory: `$room_dir/team/$role_plural/$speaker_name/`
3. Create sub-directories: insights/, advice/, connections/, concerns/
4. Create PROFILE.md with frontmatter:
   ```yaml

   name: {display_name}
   role: {speaker_role}
   affiliation: (unknown -- pending research)
   first_meeting: {today's date}
   meetings_attended: 1
   created_by: file-meeting
   research_status: pending
   ---
   # {display_name} ({speaker_role})
   ## Context
   [To be filled after proactive research]
   ## Expertise
   [To be inferred from meeting contributions]
   ## Key Contributions
   [Auto-populated from filed artifacts]
   ```
5. Echo the created path to stdout
6. Exit 0 on success, exit 1 on failure (e.g., directory already exists)

Use `set -euo pipefail`. Include `--help` flag. Make executable.

Create `scripts/research-speaker` -- a bash script that performs proactive web research on a newly identified meeting speaker, in the context of the project/room.

This implements the locked CONTEXT.md decision: "When a new profile is created, a proactive agent researches the person online in context of the project/room and builds a Data Room-specific profile. User confirms before finalizing."

**Input:** 4 arguments:
- $1 = room_dir (path to the room, e.g., `room/`)
- $2 = speaker_slug (e.g., `tyler-chen`)
- $3 = display_name (e.g., `Tyler Chen`)
- $4 = project_context (brief description of the venture/project, extracted from room/STATE.md or passed by file-meeting)

**Output:** Writes a research summary to stdout (NOT directly to PROFILE.md -- the calling command presents to user for confirmation first). Exit 0 on success, exit 1 on failure.

**Script flow:**
1. Validate inputs (4 args required, room_dir exists, speaker profile exists at expected path)
2. Read the speaker's existing PROFILE.md to get role and first_meeting date
3. Read `room/STATE.md` (if exists) to extract venture name, stage, and domain for search context
4. Construct search queries -- 3 targeted searches:
   - `"{display_name}" {venture_domain}` -- find the person in project-relevant context
   - `"{display_name}" {role} linkedin OR crunchbase OR twitter` -- professional profile
   - `"{display_name}" {venture_keywords}` -- domain-specific contributions
5. Execute searches using WebFetch or the available search tool:
   - Use `curl -s "https://www.google.com/search?q=..."` with URL encoding as a fallback
   - Parse results for: current role/company, relevant expertise, publications, social profiles
   - If no search tool is available, output a message: "Web search not available. Populate manually or configure a search MCP."
6. Compose a research summary in markdown format:
   ```markdown
   ## Research: {display_name}
   **Searched:** {date}
   **Context:** {project_context}

   ### Professional Background
   {findings or "No public information found"}

   ### Relevance to Project
   {how this person connects to the venture domain}

   ### Online Presence
   {links found, if any}

   ### Suggested Profile Updates
   - affiliation: {found affiliation or "unknown"}
   - expertise_areas: [{inferred from research}]
   ```
7. Output the research summary to stdout
8. If `--apply` flag is passed (user confirmed), update the PROFILE.md:
   - Replace `affiliation: (unknown -- pending research)` with found affiliation
   - Fill in ## Context and ## Expertise sections
   - Update `research_status: pending` to `research_status: completed`
   - Add `research_date: {today}` to frontmatter

**Error handling:**
- No internet / search fails: Output "Research unavailable" message, exit 0 (not a fatal error)
- Speaker profile doesn't exist: Exit 1 with error message
- Already researched (research_status: completed): Skip with message "Already researched on {date}"

Use `set -euo pipefail`. Include `--help` flag with usage examples. Make executable.

**Important:** This script is called by file-meeting AFTER the entire filing pipeline completes (not during). The output is presented to the user for confirmation. Only when user confirms does the `--apply` flag get used to write to PROFILE.md. This follows Pitfall 7 from research: never block the filing pipeline with research.
