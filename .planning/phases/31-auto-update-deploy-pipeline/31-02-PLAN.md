Create `lib/core/exports-log.cjs` -- manages `.exports-log.json` in the room directory:
- `readExportsLog(roomDir)` -- reads and parses `roomDir/.exports-log.json`, returns `{ deployments: [] }` if missing
- `logDeployment(roomDir, entry)` -- appends entry to log. Entry shape: `{ url, timestamp (ISO), host: "vercel", sections: ["all" or section names], private: bool, project_name }`
- `getLatestDeployment(roomDir)` -- returns most recent entry or null
- File is JSON array under `deployments` key
- Zero npm deps (fs + path only, existing codebase pattern)

Create `scripts/publish-ops` -- Bash script with subcommands (follows git-ops pattern):

**Subcommand: `check-vercel`**
- Check if `vercel` CLI is installed: `command -v vercel`
- If missing, print: "Vercel CLI not found. Install with: npm i -g vercel"
- If present, check if logged in: `vercel whoami 2>/dev/null`
- Print JSON: `{"installed": bool, "logged_in": bool, "user": "name_or_null"}`

**Subcommand: `link <room_path>`**
- Check if `room_path/exports/presentation/.vercel` exists (already linked)
- If not linked: run `vercel link --yes --cwd <room_path>/exports/presentation/`
- Print result (linked project name or error)

**Subcommand: `deploy <room_path> [--sections sec1,sec2] [--private]`**
- If `--sections` provided:
  1. Create a temp directory
  2. Run `generate-presentation.cjs <room_path> --output <temp_dir> --sections sec1,sec2` (this flag needs to be passed through -- see note below)
  3. Actually, generate-presentation does not support --sections yet. Instead: copy `exports/presentation/` to temp dir, then FILTER: read graph.json, remove nodes/edges not in specified sections, rewrite graph.json. For each HTML file, this is enough since templates render from ROOM_DATA which is embedded. So the REAL approach: run generate-presentation with a filtered room. Create a temp room copy with only the specified section dirs, STATE.md, MINTO.md, graph.json. Then generate from that. OR simpler: just copy exports/presentation/ and strip sections from the embedded ROOM_DATA JSON in each HTML file using sed/node.

  SIMPLEST CORRECT APPROACH for --sections:
  1. Copy exports/presentation/ to a temp dir
  2. For each .html file in temp dir, use node to: parse the ROOM_DATA JSON embedded in the file, filter `sections` array to only include named sections, filter `artifacts` to only those in allowed sections, filter graph nodes/edges to only those in allowed sections, rewrite the file
  3. Deploy from temp dir
  4. Clean up temp dir

- If `--private` provided:
  1. Generate a random 12-char password: `node -e "console.log(require('crypto').randomBytes(6).toString('hex'))"`
  2. Add Vercel password protection via `vercel env add PASSWORD_PROTECTION` or simpler: create a `vercel.json` in the deploy dir with `{ "authentication": { "password": "<generated>" } }`
  Actually, Vercel password protection on free tier uses "Vercel Authentication" which requires Pro plan. For free tier, implement client-side password gate: inject a password prompt page that checks against a hash stored in the HTML. Not bulletproof but functional for MVP.

  FREE TIER --private approach:
  1. Generate a password, show it to the user
  2. Hash it with SHA-256
  3. Inject a password gate `<div>` at the top of each HTML file that prompts for password, checks SHA-256 hash in JS, hides gate and shows content on match. Store hash in the HTML. This is client-side only -- not cryptographically secure but sufficient for "don't let casual visitors see this"
  4. Log the password in .exports-log.json so user can retrieve it

- Standard deploy (no flags): `vercel --yes --cwd <deploy_dir>` (deploy_dir = exports/presentation/ or temp filtered dir)
- After deploy: capture URL from vercel stdout, call exports-log.cjs to log it
- After deploy: ensure git commit + push so Vercel auto-deploy stays wired (SYNC-03): `git-ops commit <room_path> exports/presentation/ "deploy: publish presentation" && git-ops push <room_path>`

**Subcommand: `domain <room_path> <domain>`**
- Run `vercel domains add <domain> --cwd <room_path>/exports/presentation/`
- Print result

Key implementation notes:
- All vercel commands use `--yes` flag for non-interactive operation
- publish-ops is a Bash script (not .cjs) following git-ops pattern
- The script uses `set -euo pipefail` and proper error handling
- For section filtering, use a small inline node script (not a separate file) to modify the ROOM_DATA JSON in HTML files

Command body -- Larry guides the user through deployment. The command handles THREE modes:

**Mode 1: First-time setup (`/mos:publish` with no Vercel project linked)**

Flow:
1. Read `references/personality/voice-dna.md` for Larry's voice
2. Check Vercel CLI: `bash scripts/publish-ops check-vercel`
3. If not installed: guide install (`npm i -g vercel`), then `vercel login`
4. If not logged in: guide login (`vercel login`)
5. Check if room has exports/presentation/: if not, run `node scripts/generate-presentation.cjs $ROOM_DIR` first
6. Link project: `bash scripts/publish-ops link $ROOM_DIR`
7. First deploy: `bash scripts/publish-ops deploy $ROOM_DIR`
8. Show the deployed URL to the user
9. Offer custom domain: "Want to add a custom domain? Give me the domain name."
10. Explain: "From now on, every time you work in this room and push to GitHub, the site updates automatically. That's SYNC-03 -- git push triggers Vercel auto-deploy."
11. Log deployment via exports-log

**Mode 2: Selective publish (`/mos:publish --sections market-analysis,solution-design`)**

Flow:
1. Parse the --sections flag to get section list
2. Confirm with user: "Publishing only: market-analysis, solution-design. Other sections will be excluded from the deployed site."
3. Run: `bash scripts/publish-ops deploy $ROOM_DIR --sections market-analysis,solution-design`
4. Show deployed URL
5. Log deployment with sections list

**Mode 3: Private publish (`/mos:publish --private`)**

Flow:
1. Explain: "I'll add a password gate. Anyone visiting the URL will need this password to see the content."
2. Run: `bash scripts/publish-ops deploy $ROOM_DIR --private`
3. Show the generated password to the user prominently
4. Warn: "Save this password -- I've logged it in .exports-log.json but you should share it separately with your audience."
5. Log deployment with private: true

**Combined: `/mos:publish --sections X --private`** -- both flags work together.

**Subsequent runs (`/mos:publish` with project already linked):**
- Skip setup steps 1-6
- Just deploy: `bash scripts/publish-ops deploy $ROOM_DIR`
- Show URL (may be same as before if Vercel project unchanged)
