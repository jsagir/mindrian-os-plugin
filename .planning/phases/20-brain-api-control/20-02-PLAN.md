Table: brain_api_keys
```sql
id            uuid PRIMARY KEY DEFAULT gen_random_uuid()
api_key       uuid UNIQUE NOT NULL
email         text NOT NULL
name          text NOT NULL DEFAULT ''
plan          text NOT NULL DEFAULT 'free' CHECK (plan IN ('free', 'pro', 'admin'))
is_active     boolean NOT NULL DEFAULT true
expires_at    timestamptz          -- NULL = permanent
created_at    timestamptz NOT NULL DEFAULT now()
last_used_at  timestamptz
request_count integer NOT NULL DEFAULT 0
created_by    text NOT NULL DEFAULT 'jonathan'
```

Table: brain_access_requests
```sql
id          uuid PRIMARY KEY DEFAULT gen_random_uuid()
email       text NOT NULL
name        text NOT NULL DEFAULT ''
message     text
created_at  timestamptz NOT NULL DEFAULT now()
reviewed    boolean NOT NULL DEFAULT false
```

Supabase REST patterns (from auth.cjs):
```javascript
// Headers for all requests:
headers: {
  'apikey': SERVICE_KEY,
  'Authorization': `Bearer ${SERVICE_KEY}`,
  'Content-Type': 'application/json',
  'Prefer': 'return=representation',  // for POST/PATCH to return rows
}
// POST = INSERT, PATCH = UPDATE, GET = SELECT
// Filters via query string: ?email=eq.user@example.com
// Ordering: ?order=created_at.desc
// Select columns: ?select=email,name,plan,is_active
```

  <action>
Create `mcp-server-brain/brain-admin.cjs` as a standalone Node.js CLI script with `#!/usr/bin/env node` shebang and `'use strict'`.

**Zero dependencies** — use Node 20 native `fetch` and `crypto.randomUUID()`. Load .env manually with a 10-line parser (read file, split lines, set process.env for non-# lines that aren't already set). Per D-04, this is NOT a plugin command.

**Supabase REST helper** — single `supa(method, path, body)` function using native fetch with the 4-header pattern from auth.cjs (apikey, Authorization, Content-Type, Prefer: return=representation). For GET requests, omit body.

**Arg parsing** — parse `process.argv` into command + flags object. Support `--email`, `--days`, `--plan`, `--name` flags. No external arg parser needed.

**Commands (per D-05):**

1. `create --email X --name "Name" [--days N] [--plan free|pro|admin]`
   - Generate UUID key via `crypto.randomUUID()`
   - Calculate expires_at: `new Date(Date.now() + days * 86400000).toISOString()` or null if no --days (permanent)
   - Default --days to 30 per D-03. Default --plan to 'free'.
   - POST to `brain_api_keys` with all fields
   - Print the created key prominently — this is the only time the full key is shown

2. `revoke --email X`
   - PATCH `brain_api_keys?email=eq.{email}&is_active=eq.true` with `{ is_active: false }`
   - Confirm how many keys were revoked

3. `extend --email X --days N`
   - First GET current key to read current expires_at
   - Calculate new expires_at from current (or from now if expired)
   - PATCH with new expires_at
   - Confirm new expiry date

4. `list`
   - GET `brain_api_keys?select=email,name,plan,is_active,expires_at,last_used_at,request_count&order=created_at.desc`
   - Display as formatted table with status indicators

5. `usage [--email X]`
   - GET with select including request_count, last_used_at
   - If --email provided, filter by email
   - Display usage stats

6. `requests` (for BRAIN-05 — access request notification via CLI)
   - GET `brain_access_requests?reviewed=eq.false&order=created_at.desc`
   - Display pending access requests
   - This serves as the primary notification path per research recommendation (CLI-first, email as follow-up)

**Output formatting (per D-06):**
Follow MindrianOS CLI UI patterns — use ANSI colors sparingly:
- Green (\x1b[32m) for success/active
- Red (\x1b[31m) for revoked/errors
- Yellow (\x1b[33m) for warnings/expiring soon
- Cyan (\x1b[36m) for headers/labels
- Reset (\x1b[0m) after each color

Format output as structured cards:
```
--- Brain Admin ---

  Command: create
  Email:   user@example.com
  Plan:    free
  Expires: 2026-04-25

  API Key: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
  (Save this key — it won't be shown again)

---
```

**Help text** — when no command or `help` command, print all available commands with descriptions. Per the "self-teach" requirement from specifics, each command invocation should briefly explain what it does before executing.

**Error handling:**
- Missing SUPABASE_URL or SUPABASE_SERVICE_KEY: print error and exit with code 1
- Missing required flags (--email for create/revoke/extend): print usage and exit
- Supabase errors: print the error response body

Make the file executable (`chmod +x` or document in comments).
