  <name>Task 1: Add /setup meetings subcommand to setup.md</name>
  <files>commands/setup.md</files>
  <action>
Add a new section at the end of commands/setup.md for `/mindrian-os:setup meetings`. Follow the exact same structure as the existing brain and transcription subcommands (Setup, Flow with numbered steps, Important Rules).

The meetings subcommand flow:
1. **Detect Existing Configuration** -- Check .mcp.json for existing meeting source keys (read-ai, vexa, recall-ai). If found, show current config and offer to reconfigure.
2. **Ask Which Source** -- "Which meeting tool do you use?" with three options:
   - Read AI (most common -- automatic meeting notes)
   - Vexa (open-source, self-hosted option)
   - Recall.ai (enterprise API)
3. **Configure Based on Choice** -- Write the appropriate .mcp.json entry:
   - Read AI: `{"type": "http", "url": "https://api.read.ai/mcp/"}` -- OAuth handled by MCP transport layer
   - Vexa: `{"command": "npx", "args": ["-y", "mcp-remote", "https://api.cloud.vexa.ai/mcp"], "env": {"VEXA_API_KEY": "{key}"}}` -- collect API key first
   - Recall.ai: `{"command": "npx", "args": ["-y", "@anthropic/recall-mcp"], "env": {"RECALL_API_KEY": "{key}"}}` -- collect API key first
   Follow the same .mcp.json merge pattern as brain/transcription (read existing, merge, write back).
4. **Test Connection** -- Call the meeting source's list-sessions/list-meetings tool to verify connectivity. On failure, offer to remove config and retry.
5. **Confirm** -- "Meeting source connected. Now use `/mindrian-os:file-meeting --latest` to grab your most recent meeting."

Important Rules (same pattern as other subcommands):
- Never echo API keys back in conversation
- Never write credentials to the plugin directory
- .mcp.json goes in workspace root
- If .mcp.json already has other configs, merge -- do not overwrite
- Remind user about .gitignore

Keep the `--join <url>` stub as "Not yet available. Coming in a future update." (that is a v3.0 feature).
