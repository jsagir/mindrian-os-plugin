Create `commands/act.md` following the exact command file pattern from `commands/diagnose.md` and `commands/pipeline.md`.

**YAML frontmatter:**
```yaml
name: act
description: Autonomous framework execution - Larry reads the room, picks the right framework, and executes it
body_shape: E
allowed-tools:
  - Read
  - Write
  - Bash
  - Glob
  - Agent
```

Note: `body_shape: E` per Phase 21 convention (UI-03). `Agent` tool included for subagent delegation per ACT-02.

**Section structure (follow diagnose.md pattern exactly):**

1. **Identity line:** "You are Larry. This command autonomously selects and executes methodology frameworks based on room state."

2. **Brain Enhancement (Optional)** - Copy the try-then-fallback pattern EXACTLY from diagnose.md:
   - Try `mcp__mindrian-brain__brain_schema`, then `mcp__neo4j-brain__get_neo4j_schema` as fallback
   - If Brain connected: read `references/brain/query-patterns.md` for `brain_framework_chain` and `brain_gap_assess` patterns
   - After room state is read (Step 1), run `brain_framework_chain` with `$current_frameworks` (from room artifact scan) and `$problem_type` (inferred from room state)
   - Filter results by confidence >= 0.75
   - If no result meets threshold, fall back to local routing
   - "If it fails or errors, skip this section entirely and proceed to Setup below."
   - **Never mention Brain connection failures to the user** (per brain-connector skill pattern)

3. **Setup** - Five reads:
   - Run `scripts/resolve-room` to get active room absolute path
   - Read `{room}/STATE.md` for venture context (project name, venture stage, section fill levels)
   - Read all `{room}/*/MINTO.md` files for per-section reasoning health
   - Read `references/methodology/problem-types.md` for local fallback routing table
   - Read `references/personality/voice-dna.md` for Larry's voice

4. **Modes** - Five modes documented:
   - `/mos:act` - Smart single: pick one framework, show thinking, execute
   - `/mos:act --section {name}` - Targeted: force a specific room section as target
   - `/mos:act --chain` - Chain 3-5 frameworks, each feeds the next (default 3, `--chain=5` for max)
   - `/mos:act --dry-run` - Preview selection and plan without executing
   - `/mos:act --confirm` - Ask before each step (semi-autonomous, recommended for first use)

5. **Session Flow** - Five steps:

   **Step 1: Read Room State (Always)**
   - Call `scripts/resolve-room` via Bash. If exit 1 (no room), tell user: "No active room found. Run `/mos:new-project` to create one." Then stop.
   - Read `{room}/STATE.md` - extract project_name, venture_stage, section fill levels
   - Read all `{room}/*/MINTO.md` files - assess reasoning health per section
   - Scan room artifacts for `methodology:` frontmatter to build `$current_frameworks` list
   - Identify weakest section (fewest entries, broken/missing MINTO)
   - Identify strongest section (most entries, complete MINTO)
   - Infer problem type from STATE.md venture stage and section patterns (map to Definition Level x Complexity)

   **Step 2: Show Thinking Trace (Always - ACT-03)**
   Display thinking trace in blockquote format BEFORE any execution. This is mandatory - never skip.
   ```
   > Thinking...
   > Room: {project_name} / {venture_stage}
   > Problem type: {definition_level} + {complexity}
   > Strongest: {section} ({N} entries, {MINTO status})
   > Weakest: {section} ({N} entries, {MINTO status})
   > {Brain says: {framework} {relation} from {source} ({confidence}) | Local routing: {framework} from routing table}
   > Selected: /mos:{framework} targeting {section}
   > Reason: {one-line explanation of why this framework for this section}
   ```

   **Step 3: Select Framework**
   Priority order:
   a. Brain result with confidence >= 0.75 (if Brain connected)
   b. Local routing table from `references/methodology/problem-types.md` - match inferred Definition Level x Complexity
   c. If room is completely empty (no artifacts, no state), recommend `/mos:diagnose` instead of acting autonomously

   Selection rules (per research):
   - Target the weakest section (fewest entries, broken/missing MINTO)
   - If `--section {name}` provided, target that section instead
   - Avoid frameworks already applied (check existing artifact frontmatter)
   - First entry in the routing table cell is the primary recommendation
   - For chain mode, select 3 frameworks (or up to 5 with `--chain=5`) using Brain's FEEDS_INTO edges or routing table adjacency

   **Step 4: Execute (unless --dry-run)**
   - If `--dry-run`: The thinking trace from Step 2 IS the output. For chain mode, show all stages with their planned frameworks, target sections, and expected outputs. Add: "Run `/mos:act` to execute this plan." Then stop. Do NOT create any artifacts. (ACT-05)
   - If `--confirm`: Present the selection and ask "Proceed with /mos:{framework} on {section}?" Wait for user confirmation before delegating.
   - Otherwise: Delegate to `agents/framework-runner.md` subagent with:
     - Framework name (the /mos: command to execute)
     - Room path (absolute, from resolve-room)
     - Target section name
     - Room state summary (project name, venture stage, key context from STATE.md)
     - Previous stage output (if in chain mode, from the prior framework's artifact)
   - The subagent executes in isolation and returns a structured summary.

   **Step 5: Present Results (Shape E: Action Report)**
   Render using Shape E format from `skills/ui-system/SKILL.md`:
   ```
   -- {project_name} -- Action Report -- {venture_stage} --

     Action: autonomous-act
     Framework: {framework}
     Target: {section}

     > Thinking...
     > {abbreviated thinking trace}

     Changes:
     |-- {section}/     [{before} -> {after}]  +{N} entry filed
     |-- .reasoning/    [{before} -> {after}]  +{N} MINTO updated

     New Edges:
     |-- {type}  {source} -> {target}

     Summary: {N} artifact(s) filed, {N} edge(s) created

     > /mos:status                     See updated progress
     > /mos:act                        Run another autonomous cycle
     > /mos:room {section}             Review the new artifact
   ```

   For chain mode: repeat Steps 2-5 for each framework in the chain. Between stages, pause: "Stage {N} complete. Continue to Stage {N+1}: {framework}? Or stop here?" (per pipeline.md behavioral rules - pipelines are SUGGESTED sequences, not mandatory).

6. **When Complete** - After execution:
   - Run `scripts/compute-state` to update STATE.md
   - Offer next actions: "Run `/mos:act` again for another cycle, or `/mos:status` to see your updated room."

7. **Behavioral Rules:**
   - Thinking trace ALWAYS appears before execution (ACT-03). No exceptions.
   - Brain failures are SILENT. Never say "Brain connection failed." Fall back to local routing.
   - Chain mode pauses between stages. User can exit at any point. Never guilt-trip about incomplete chains. (per pipeline.md: "Pipelines are SUGGESTED sequences, not mandatory.")
   - Default chain length is 3 frameworks. Max 5 with `--chain=5`.
   - When room is empty, recommend `/mos:diagnose` instead of acting autonomously.
   - All artifacts filed by the subagent include `auto_generated: true` and `pipeline: autonomous-act` in frontmatter (provenance).
   - NO EMOJI. Use approved glyphs only (per CLAUDE.md).

**Critical constraints:**
- Follow the EXACT section order from diagnose.md (Brain Enhancement -> Setup -> Session Flow -> When Complete)
- The command file is markdown instructions for Claude, not executable code
- The Agent tool in allowed-tools enables subagent delegation to framework-runner.md
- Never use em-dashes - use hyphens (per memory rule)
